1、	绪论
1.1	研究背景与研究意义
1.1.1	浅谈大语言模型
近年来，随着大语言模型（Large Language Model, LLM）的深入发展，大语言模型不再仅仅局限于传统的自然语言处理（Natural Language Processing, NLP）任务。借助工具调用（Function Calling）机制与模型上下文协议（Model Context Protocol, MCP），大语言模型成功跨越了纯文本交互的鸿沟，具备了与外部环境交互及执行复杂任务的能力；借助检索增强生成 （Retrieval Augmented Generation, RAG），大语言模型突破了预训练数据的静态限制，获得了动态获取并整合外部垂直领域知识的能力，有效抑制了事实性幻觉；借助记忆机制 （Memory），大语言模型得以沉淀历史交互经验与情境上下文，实现了从单次、无状态对话向长期连贯、具备个性化认知的智能体演进。
大模型与智能体技术的快速演进，同样在游戏行业掀起了重大的技术变革。在工业界，大语言模型正被广泛应用于重塑非玩家角色（None-Player Character, NPC）的交互体验与动态行为逻辑。例如，网易在《逆水寒》手游中实装了基于大语言模型的智能NPC系统，使游戏角色能够根据玩家的自然语言输入与当前环境，生成不拘泥于预设脚本的对话与行为反馈；育碧（Ubisoft）推出的 NEO NPC 原型项目，则深入探索了生成式AI在游戏叙事与角色情感深度上的潜力。
现阶段，相较于工业界侧重于 NPC 交互体验的优化，学术界与前沿探索者则将研究重点更多地聚焦于涉及复杂任务调度、资源管理与多智能体协作的模拟（Simulation）游戏品类上。 例如，Park 等人的工作（Generative Agents）构建了一个类似《模拟人生》（The Sims）的沙盒环境，探索了智能体社群的涌现性社会行为；Gong 等人的研究（MindAgent）则在简化的《胡闹厨房》（Overcooked）环境中，验证了多智能体在多步骤任务中的规划与协作能力；而 Wang 等人（Voyager）更是直接利用《我的世界》（Minecraft）作为测试环境，展示了基于大模型的智能体在开放世界中从零开始获取资源与制造物品的终身学习能力。
1.1.2	模拟类游戏
模拟类游戏是通过电脑对真实世界中某些特定场景或领域进行仿真，使玩家能够在一个具有动态变化的交互系统中，来实现娱乐或者练习等目的的游戏。而模拟经营属于这一品类中的重要分支，要求玩家以管理者（有时也同时担任参与者）的视角，来对游戏中的资源进行统筹分配，对游戏中的产线和虚拟非玩家角色（NPC）进行调度管理，来实现特定的目标。
在传统的模拟经营类游戏中，NPC 的行为决策系统普遍采用基于规则的实现方案，其中最具代表性的是有限状态机（Finite State Machine, FSM）与行为树（Behavior Tree, BT）。但这种方法存在许多缺陷，核心可归纳为以下三点：
第一，基于静态优先级的任务调度机制，无法应对复杂环境的动态变化。基于规则的 NPC 系统中，任务的执行顺序、权重分配、触发条件均由人工预先固化设定，仅能响应预设范围内的单一变量变化。即使许多游戏支持让玩家自己设置优先级来调整整体策略，NPC依旧无法实现直接根据当前环境动态选择当前优先执行的任务。
第二，交互维度与深度存在原生局限，NPC的行为组合空间被限制在人工预设的逻辑边界内。虽然无论哪种方法，NPC的行为都受制于游戏规则下，只能执行规则中基本行为。但是，基于规则的方法的最大缺陷在于：这些原子操作之间的组合方式与触发逻辑，是完全依赖人工穷举和硬编码脚本。当玩家在游玩过程中出现了游戏策划未曾想到的场景，NPC便无法给出合理的回应，就会产生割裂感进而破坏玩家的沉浸感。
第三，由编码的脚本控制的行为逻辑，导致玩法迭代与内容拓展的边际成本极高。为了尽可能地覆盖玩家可能触发的各种场景，策划与开发人员必须投入大量人力去设计和编写极其庞杂的状态机或行为树节点。这种强耦合的底层架构不仅维护困难，而且每新增一项游戏机制或一类 NPC 职能，都需要对现有的逻辑树进行牵一发而动全身的重构与测试，极大地限制了模拟经营类游戏的开发效率与内容体量上限。
1.1.3	模拟经营游戏结合大语言模型的优势
有研究指出，随着参数规模的扩大，大语言模型展现出了小模型所不具备的、且无法通过简单规模外推来预测的涌现能力（Emergent Abilities）。依托于这种强大的底层推理与泛化能力，Park 等人通过建立 Smallville 虚拟沙盒小镇与生成式智能体（Generative Agents），成功验证了在模拟环境中，多个智能体之间的自然交互能够自发交织出复杂的、涌现性的社会行为（Emergent Social Behaviors）。以上研究充分说明了，将大语言模型引入模拟经营游戏的底层架构，能够从根本上突破传统规则 AI 的原生局限，其优势具体体现在以下三个维度：
第一，摆脱静态优先级队列，实现基于环境感知的动态任务调度。大语言模型所具有的常识推理能力可以使NPC不再受限于人工固化的执行顺序与权重分配，例如Yiding Wang等人提出的D2A方法通过在底层构建包含社交、生理、自我实现等多维度的动态价值系统，赋予智能体类似于人类的需求理论。在此基础之上 ，NPC 就可以不再受限于人工固化的执行顺序，而是能够实时评估自身的欲望状态（Desire States），并自发地提出、评估和选择当前最能满足其内在需求的行为。这种由内而外的驱动模式，使得智能体在面对复杂多变的游戏环境时，能够展现出高度符合逻辑、且极具适应性的拟真日常行为，更从根本上为玩家提供了深度的沉浸式生态体验。
第二，突破硬编码的逻辑边界，组合游戏规则内的行为产生涌现式交互。区别于有限状态机穷举所有可能性的做法，基于大语言模型的 NPC可以通过将游戏内NPC可以做的行为抽象为可以调用的工具（Tools）或者接口（API）。面对策划未曾预料到的复杂边缘场景，大语言模型扮演的NPC能够根据当前的游戏环境和角色描述等上下文信息，将基础行为进行动态拼装与自由组合来应对复杂场景。这种基于意图驱动的开放组合空间，彻底打破了传统 AI 机械僵化的决策边界，极大地提升了玩家交互的自由度与沉浸感。
第三，重塑底层开发管线，显著降低复杂系统的迭代与维护成本。 引入大语言模型后，游戏开发者只需要通过自然语言提示词定义好 NPC 必须遵守的游戏规则、角色人设等基础信息，并提供标准化的环境交互接口（API/Tools），无需再人工去穷举并验证所有的状态节点跳转。这种方法利用大语言模型将高层的行为决策与底层的状态控制彻底解耦，能将开发者从繁琐的逻辑纠错与状态机维护中彻底解放出来，使其能够将核心精力聚焦于玩法机制的设计与内容扩充，极大地提升了包括模拟经营类游戏在内、多种依赖复杂 NPC 生态的游戏品类在长线研发中的工程效能与迭代速度。
1.2	面临的问题与挑战
诚然，将大语言模型引入游戏确实能有力地打破状态机和行为树的限制，赋予智能体NPC极高的自主性和涌现能力，并显著降低NPC迭代成本。然而，从理想的沙盒实验室环境走向真正商业化、具有复杂交互的游戏产品 时，大模型在底层架构与工程落地方面仍面临着诸多严峻的挑战。具体而言，主要集中在以下几个维度：
第一，不可避免的幻觉（Hallucination）问题。 LEI HUANG等人的综述指出，由于数据、训练和推理过程中的缺陷，现有大模型不仅会产生违背客观设定的事实性幻觉（Factuality Hallucination），更会爆发出不忠实于指令与上下文的忠实性幻觉（Faithfulness Hallucination）。在严密的底层运行逻辑中（尤其是对于包含复杂生产链路与长线资源管理的模拟经营类沙盒），后者往往是致命的。近期利用游戏环境进行评估的研究充分暴露了这种忠实性幻觉的严重性：Wenye Lin等人的GameBOT基准测试发现，所有被测模型在复杂游戏中的中间推理步骤平均得分都相对较低，且规则越复杂，逻辑断裂就越明显；Dawei Wang等人的TowerMind工作也证实，幻觉会随关卡难度提高而加剧，导致大模型输出的合法动作比例下降，且极易做出缺乏长线规划、浪费资源的盲目决策。因此，如何降低大语言模型对游戏规则的幻觉、提高大语言模型输出指令的准确率、增强大语言模型对动作的利用和理解，将会是将大语言模型应用在控制游戏NPC的关键挑战。
第二，现有的大模型推理和响应速度难以满足游戏的实时性要求。Artificial Analysis网站的评测数据显示，目前主流的闭源模型，如Gemini 3 Pro、Qwen3.5、GLM-5等模型在输出500个Tokens的端到端响应时延往往都在20秒以上。本文实验评估表明，本论文中的游戏环境从大语言模型处获得生成的指令往往需要2-3秒钟左右。这意味着如果游戏和大语言模型的生成采用异步的方式，大语言模型所获得的游戏状态往往是过时的。而如果采用同步的方式，则玩家经常需要等待大模型生成指令，这严重不符合游戏的实时性要求。
	此外，还有不少研究也将目光放在了减少 Tokens 的消耗上。Park 等人提出的生成式智能体（Generative Agents），仅模拟 2 天的游戏内时间、运行 25 个智能体，就产生了数千美元的开销（Lyfe Agents 论文对其保守测算，总成本至少 2000 美元，折算为实时交互场景下，单智能体每真人小时成本高达 25 美元）。而 Zhao 等人的 Lyfe Agents，则尝试通过类脑启发的轻量化架构设计，从决策、监控、记忆三大核心环节大幅削减非必要的大语言模型（LLM）调用，同时优化 prompt 长度、降低无效 tokens 消耗的方法，成功地将运行成本降低至每个智能体每真人小时 0.5 美元左右，较现有主流方案实现了 10-100 倍的成本下降，但面向大规模商业化落地仍有优化空间。

2、	国内外相关研究
2.1	游戏AI发展
自南梦宫的《吃豆人》（Pac-Man）作为首款引入初级AI的游戏问世以来，游戏AI技术便始终与电子游戏产业的演进休戚与共。时至今日，人工智能已成为各大游戏引擎的核心竞争力与重要卖点。整个行业也在不断探索，致力于让非玩家角色（NPC）具备媲美真人的行为逻辑，从而对玩家的复杂交互做出更丰富、更拟真的动态反馈。
Ian Millington在其《游戏中的人工智能》（AI for Games）一书中总结的游戏AI模型将AI任务分为三个部分：移动（Movement），决策（Decision）和策略（Strategy）。其中移动和决策都是对于单个角色的AI，而策略则是指一组角色群体整体所采用的方法。
移动负责将上层的行为决策，转化为底层的执行，主要是角色的空间位置的变化。移动主要关注两个方面：一、基本的移动算法，如速度、转向、避障等。二、路径发现和规划算法，常见的技术有迪杰斯特拉算法（Dijkstra）、A*算法和导航网格算法（NavMesh）。
决策则主要负责计算角色下一步的具体行为，常见的方法有有限状态机（Finite State Machine， FSM）、行为树（Behavior Tree）和面向目标的行为规划（Goal-Oriented Action Planning，GOAP）。FSM和行为树作为经典方法，目前在市面上的游戏广泛使用，许多游戏引擎（如Unreal）都在底层提供了完善的系统。
策略则负责通过AI算法控制由一组角色构成的团体的行为。团队中的角色可能拥有特色的移动和决策，但会受到整体策略的影响。常见的算法有协助环境感知的影响图（Influences Maps）、控制群体移动和阵型的集群算法。
近年来，随着机器学习、深度学习的发展，许多研究者尝试在对抗性游戏中使用强化学习（Reinforcement Learning）等前沿技术，来训练甚至能够超越人类顶尖水平的AI智能体（例如《星际争霸》中的AlphaStar、《Dota 2》中的OpenAI Five和《王者荣耀》的绝悟）。
2.2	大语言模型与游戏AI
2017年Vaswani等人提出了Transformer架构，打破了传统卷积神经网络（Convolutional Neural Network, CNN）和循环神经网络（Recurrent Neural Network, RNN）的限制，彻底改变了NLP领域的发展，催生出了以GPT为代表的大语言模型。
尽管受限于数值计算与空间几何认知，大语言模型目前尚不适合直接处理底层的移动逻辑，但其强大的常识推理与文本生成能力已引起学界的广泛关注。目前，主流的研究方向正积极探索如何跨越这些限制，重点利用大语言模型来强化游戏中NPC的决策机制。
Joon Sung Park等人的Generative Agents工作明确指出，利用大语言模型扮演游戏角色，能发挥大语言模型的涌现能力，在智能体社会内出现涌现的社会行为，这是传统基于规则的决策方法所无法做到的。同时，相较于基于学习的方法依赖对抗性较强的游戏环境提供奖惩机制，基于大语言模型的方法更适合于角色扮演、模拟类游戏中。
而在实际工业领域，已经有不少实验性质的作品。其中最引人注目的当属Anuttacon团队推出的AI互动类游戏《群星低语》（Whispers from the star）。该游戏以大语言模型和AI语音交互为核心玩法，彻底打破了传统游戏中“预设对话树”的限制。游戏主角能够实时理解玩家的自然语言输入，其所有的情绪反馈、动作反应乃至整个世界剧情的走向，均由AI实时演算生成。
同时，游戏作为一个复杂环境，也可以作为大模型的测试平台。
Wenye Lin等人的GameBoT测试基准利用包括棋类、动作类（如Pong）、纸牌类（如德州扑克）以及博弈论任务在内的动态竞技游戏环境，通过模型与模型之间对战的方式来评估模型能力。这种方式充分利用了游戏中动态交互所构成的问题空间，有效地缓解了传统静态测试集容易引发的数据污染问题。同时其采用了结合人类专家知识的思维链（Chain of Thought, CoT）来指导模型进行思考，并对思考过程和中间步骤进行评估，来全面地说明模型的能力
Dawei Wang等人则利用了实时策略类（RTS）游戏对长期规划和决策能力的要求，使用RTS子类塔防游戏环境，提出了TowerMind评估环境，设计了 5 个基准关卡，在多模态输入设置下评估主流 LLM，发现 LLM 与人类专家在能力、幻觉维度存在显著性能差距，且暴露 LLM 规划验证不足、决策缺乏多终局思维、动作利用低效等问题。
而Hao Kang等人则针对高频交易、实时竞技游戏等对响应速度有严格要求的场景，研究了大模型在延迟敏感型任务中的延迟和精度权衡问题，并提出了以《街头霸王》（StreetFighter）游戏为测试环境的全新的延迟敏感型评估基准StreetFighter，发现在实时动态环境中，延迟与精度之间的权衡直接决定了 LLM 智能体的性能表现。
2.3	智能体
近年来，大语言模型的发展日新月异。研究人员逐渐发现，大语言模型的能力已不再局限于单一的自然语言生成，其强大的指令遵循与逻辑推理能力使其能够通过输出结构化内容来调用外部工具（Tool Learning）。同时，为了突破模型在特定领域知识和长期上下文处理上的瓶颈，研究者们通过提示词工程（Prompt Engineering）、检索增强生成（RAG）以及记忆模块等技术，将会话历史与领域知识注入模型。这种由“感知、记忆、大脑（大模型）、行动”四大模块构成的系统，标志着大模型从单纯的对话机器人正式向能够执行具体生产任务的智能体演进。
而这一发展方向，正好契合了电子游戏领域对未来游戏AI技术的发展愿景。正如米哈游和复旦大学NLP实验室发表的那篇综述性论文中所构想的一样，大语言模型驱动的智能体将作为具备自主规划与生存能力的“具身实体”或“人类级参与者”进入游戏环境，利用大模型作为高级控制大脑，在动态的虚拟沙盒中进行环境感知、决策推理和技能调用。它们不仅能像人类玩家一样进行复杂任务的拆解与执行，还能通过自然语言理解他人的意图，与真实玩家及其他NPC展开深度的策略协商与协同合作。
目前，有不少研究已经在尝试构建智能体来代理游戏中的角色，其中Guanzhi Wang等人基于《我的世界》（Minecraft）开发的Voyager通过自动课程学习（Automatic Curriculum）机制和不断进化的代码技能库（Skill Library），无需人类干预即可在开放世界中进行长时间的自主探索与自我进化，充分展现了基于大模型的智能体在复杂游戏环境中的强大行动力与泛化能力。
同时，多智能体系统是智能体的一个发展方向。相较于单一智能体系统，多智能体系统可以组合多个专一于特定领域、特定任务、特定能力的智能体，来解决更为动态、复杂的任务。同时，多智能体还能允许智能体之间进行互动，来模拟真实世界中的协作、辩论、竞争等关系。
而对于大多数电子游戏而言，玩家很少只需要和一个NPC进行交互，尤其是在协作场景较多的合作闯关类游戏和管理NPC进行生产的模拟经营游戏中。《胡闹厨房》（Overcooked）就是一个学术界热门的例子。MindAgent虽然成功地将游戏转化为更适合与智能体交互的形式，并提供给智能体对游戏内的NPC操作的方法，但其本质还是通过单一智能体整合游戏信息和规则，通过输出多条指令的方式控制游戏内的智能体。而ProAgent则通过分析场景并推断队友意图，再制定高层协作规划并校验可行性，随后将合法规划拆解为环境可执行的底层动作，同时根据队友实际行为动态修正对其的意图认知，让智能体无需针对特定队友或场景预训练，实现多智能体的零样本自适应协作。

3.2	RimSpace
RimSpace 是基于 Ludeon Studios 开发的外星殖民地模拟经营游戏《边缘世界》（RimWorld），对其核心机制与底层逻辑进行抽象、提炼与简化后，所设计实现的轻量化游戏仿真环境。该环境剥离了传统游戏中复杂的视觉与冗余系统，专注为多智能体系统提供一个高度结构化、逻辑严密的测试与运行环境。
RimSpace具有以下优势：
	原子化的动作空间：RimSpace提供了极简且完备的原子动作集：Move（移动至某个地点），Take（从设施里拿取物品），Put（向设施里放入物品）、Use（使用设施，包括种植、收获、生产、进食和睡眠）和Wait（等待一定时间）。这种设计将复杂的角色行动拆分为简单的原子操作，并提供了统一的接口，方便外部多智能体系统进行控制。
	角色状态：环境内置了基础的生理状态模拟，引入了饱食度和精力值。角色必须通过在餐桌（Table）进食或在床铺（Bed）睡眠来维持基础工作能力。这种生存压力的引入，提升了角色和环境的复杂度。要求智能体在决策时不仅要明确自己的任务，还需要了解自己的状态，权衡短期生存需求与长期生产目标。
	明确的技能壁垒与角色分工：系统设定了严格的技能机制将角色划分为农夫（Farmer）、厨师（Chef）与工匠（Craft）。每个角色都有自己特有的工作能力和工作场所。这种异质化的角色设定，要求智能体对其当前扮演的角色能力有精确的认知，而不是仅仅停留在文本输出上。
	闭环的生产与物理交互逻辑：环境提供了包含培养舱（CultivateChamber）、工作站（WorkStation）、炉灶（Stove）及仓库（Storage）等在内的完整可交互设施。从原材料的种植收获，到中间产物的仓储中转，再到最终成品的加工合成，形成了一套需要高度协同的闭环生产链路。
	多层次的协作机制：环境支持复杂的多智能体协同。其中既包含基于技能壁垒的强制性依赖协作（如工匠制作衣服必须依赖农夫种植的棉花），也涵盖了旨在提升整体运转效率的非强制性软协作（如跨角色参与物品搬运与物流中转），为验证智能体之间的群体协作策略提供了充分的测试空间。
为了实现与外部智能系统的无缝对接，RimSpace 在数据接口层面进行了标准化设计。环境能够在每一帧或特定的决策节点，将当前游戏世界的全局状态（包括所有实体的位置、当前库存、角色的生理状态、各交互设施的运作阶段等）统一封装并以 JSON 格式输出。
基于上述环境特性与接口设计，RimSpace 对接入的外部驱动系统提出了明确的要求。外部 MA 系统需要能够实时解析环境输出的 JSON 状态数据，并据此满足五个核心功能需求：精准的角色与配方认知、全局的状态感知、多智能体间的任务协作，以及权衡生存与生产的自主决策能力。外部系统需作为角色的“大脑”下发具体的操作指令，从而驱动 RimSpace 环境的持续运转。
3.3	角色设定
在RimSpace中，为了实现系统的自治与分工协作，环境对智能体的行为边界进行了严格的定义。智能体既具备维持自身运转和工作的通用基础能力，又有专属于特定职业的生产能力。本环境主要包含了三类核心职业：农夫（Farmer）、厨师（Chef）和工匠（Crafter）。这三类职业都有只有特殊的生产物品和特定的工作场所。
3.3.1	基本能力
基本能力构成了智能体在 RimSpace 中生存与交互的底层动作空间。无论是何种职业的智能体，均具备以下两类基础通用能力：
	基础的物理交互能力：智能体能够执行空间位移与基础的物流操作。具体包括移动到指定的设施、从当前所在位置提取指定物品放入背包和向当前位置放入背包中的物品。这构成了跨智能体物资流转的底层基础。
	生理状态维系能力：所有智能体都具备在餐桌（Table）进食来提高自身饱食度、在床铺（Bed）睡眠来提高自身精力值的能力。
3.3.2	农夫能力
农夫（Farmer）拥有种植（CanFarm）的能力，在系统中扮演着原材料生产者的角色，是整个生产链路的源头。
拥有种植能力的角色，可以在培养舱（CultivateChamber）处进行两项工作：一、通过种植，将待种植（WaitingToPlant）的培养舱种上玩家指定的作物。二、通过收获，将可以收获（ReadyToHarvest）的培养舱中作物收获，产生对应原材料（棉花或者玉米）。
3.3.3	厨师能力
厨师（Chef）拥有烹饪（CanCook）的能力，负责系统的后勤保障，是维系所有智能体生理状态运转的核心枢纽。
拥有烹饪能力的角色，可以在炉灶（Stove）处进行工作。可以使用原材料玉米加工成可供所有角色使用的套餐。
3.3.4	工匠能力
工匠（Crafter）拥有制作（canCraft）的能力，负责系统中的原材料加工和手工制作，是实现环境经济价值增长的核心角色。
拥有制作能力的角色，可以在工作站（WorkStation）处进行手工制作任务。可以将原材料棉花进行加工为中间产物：棉布和棉线，并最终利用二者制作出终端产品衣服。
3.4	交互环境
在RimSpace中，角色无法凭空进行生产与生存，必须同游戏内预设的设施进行交互。这些可交互地点成了智能体执行具体动作的作用客体，涵盖了从底层农业生产到高级物品制造，再到角色生理状态恢复的完整场景。
3.4.1	农业与制造设施
农业与制造设施是系统中产生经济价值与物资增量的核心区域，对于使用进行生产操作，具备严格的职业准入限制。同时也允许其他职业的角色对这些设施的存储空间进行操作。
	培养舱（CultivateChamber）：用于农业种植的基础设施，仅限农夫（Farmer）使用。培养舱具有一套完整的生命周期状态机：待种植（WaitingToPlant）→ 生长中（Growing）→ 准备收获（ReadyToHarvest）。农夫通过种植将培养舱状态切换至生长阶段，并在作物成熟后通过收获提取棉花或玉米，放入培养舱的存储空间，同时将培养舱重置回待种植状态。
	炉灶（Stove）：用于食品加工的烹饪设备，仅限厨师（Chef）使用。厨师在此处执行烹饪，消耗玉米制作出维持系统运转必需的套餐食物。该设施的运转强依赖于前置原材料的充足供应。
	工作站（WorkStation）：用于手工制造与物品深加工的工作台，仅限工匠（Crafter）使用。工匠在此处执行制作，消耗指定的原材料或加工品，按照设定好的任务图纸合成更高级的物品（棉线、布料及衣服）。
3.4.2	仓储与物流节点
	仓库（Storage）：作为环境中的全局物流枢纽与物资集散中心，对所有智能体（Farmer、Chef、Crafter）开放。	
3.4.3	生理恢复设施
为了模拟真实的生存压力，环境提供了专门用于维系智能体工作能力的恢复设施，供所有角色无差别使用：
	餐桌（Table）：用于恢复角色饱食度。智能体在此处进食，每次需消耗 1 个套餐来补充能量。
	床铺（Bed）：用于恢复角色精力。智能体可以在这里睡眠，以将体力恢复至满值，确保后续生产任务的正常进行。

4、	基于黑板任务和意图规划的多智能体协同机制
4.1	多智能体整体架构
本论文所采用的多智能体主要分为以下四个主要模块：
	感知器（Perceiver）：负责接收游戏发送的JSON格式化数据，解析整个游戏的人物和环境状态，并根据游戏内的设施状态推断出玩家的高层意图，或者说玩家发布的任务，如培养舱种植物品、工作站和灶台生产对应的物品。
	黑板（Blackboard）：多智能体协作的“数据中枢”。作为共享的内存区域，它基于目标（Goal）与前置条件（Preconditions）机制，统一负责全局任务的管理、存储、校验与动态分发。
	大语言模型（LLM）：基于大语言模型，是智能体的大脑。负责结合当前游戏环境、自身状态和黑板中的任务产生高层的行为意图。通过结合特定职业的背景故事（Profile）、世界状态、角色状态和黑板任务，构建出自然语言描述的 Prompt，交由 LLM 推理并输出高层决策（如“制作物品”、“搬运资源”等）。
	规划器（Planner）：是智能体将高层意图转化为具体动作的关键。负责将 LLM 输出的抽象意图转化为具体执行流，其内部包含任务规划器 (Task Planner) 和 行为规划器 (Action Planner) 两个子模块。前者负责宏观复杂任务的前置条件校验与供应链拆解，后者负责将明确的意图序列化为底层游戏引擎可直接执行的原子动作队列。
 
	一次请求的流程如图所示，当游戏中的NPC完成动作后，会将游戏的状态连同着请求发送给运行在本地的多智能体服务器，服务器检查角色信息，如果当前还有未执行完的动作序列，则优先返回动作序列中的动作，否则就将游戏状态移交给感知器。感知器优先感知器对其进行解析，将游戏的环境信息和当前环境中隐式的任务（培养舱待种植、玩家发布了制作物品的任务等）更新至黑板。组装角色的背景故事、转化为状态描述的个人状态、黑板中的任务信息和输出格式等内容为提示词（prompt）后，交由大语言模型进行常识推理并输出高层意图。该意图会流转至规划器进行拆解，如果满足条件，则生产对应的动作队列；若遭遇资源瓶颈，则反向将其拆解为子任务并重新投递至黑板，从而驱动系统进入下一轮的多智能体协作循环。
	后续小节将对黑板任务、规划器两个核心部分的具体实现细节展开说明。
4.2	黑板
在多智能体系统中，智能体之间有多种形式的交流结构，常见的有：分层结构（Layered）、去中心化（Decentralized）、中心化（Centralized）和共享消息池（Shared Memory Pool）。本文采用的黑板（Blackboard）就是共享消息池的一种。黑板充当了全局的共享内存与任务中枢，所有智能体均不进行直接的点对点通信，而是通过向黑板发布任务或认领任务来实现信息的交流与协作。黑板内部则通过抽象的数据结构和动态校验算法，保证了全局信息与任务的有效性和准确性，其核心设计机制如下所述：
4.2.1	任务表示模型
在本文的黑板系统中，任务并非是简单的字符串文本，而是黑板任务（Blackboard Task）对象。该结构不仅包含供大语言模型理解的自然语言描述（Description），更核心的是包含了用于游戏状态校验的最终目标（Goal）、任务前置条件列表（Preconditions）以及任务技能约束（required_skill）。
其中Goal是一种为了检验游戏环境是否满足某种条件所设计的五元组（target_actor, property_type, key, operator, value）：
	target_actor：目标实体（Stove、CultivateChamber等游戏内设施或者全局库存等）
	property_type：需要检查的属性大类（如库存、种植信息等）
	key：具体的属性键值（如物品或生长阶段）
	operator 与 value：逻辑比较符（如 ==, >=, < 等）与期望的阈值。
例如，“确保全局库存中棉花数量至少为5”这一需求，可被统一抽象为 ("Global", "Inventory", "Cotton", ">=", 5)。这种声明式的表达方式，使得系统能够以统一的逻辑接口处理游戏的世界状态。
同时Goal提供了 is_satisfied() 方法，能够根据黑板中存储的世界状态检查任务目标或前置条件是否已经在当前游戏世界中满足，返回布尔结果，为后续的任务清理和准入提供客观依据。
前置条件列表则是一个Goal的数组，用于表示顺利推进该任务需要满足的前置条件。只有当Goal数组中所有的Goal都满足，才说明当前任务是可以执行的。
任务技能约束是根据当前任务所需的技能，只有具有当前技能的角色才能推进这个任务。
4.2.2	任务的管理
在黑板系统中，任务的生命周期管理（添加、清理与分发）依赖于具体的状态校验逻辑，主要包含以下三个核心处理流程：
	任务的去重机制：当感知器或规划器通过 post_task() 向黑板发布任务时，系统会通过遍历现有任务池，比对任务的目标对象（Goal）进行去重。如果发现重复，系统将直接丢弃新的任务请求并返回已有实例。此机制防止了环境轮询扫描以及多智能体并发请求所导致的同类任务重复累加。
	游戏状态更新后的任务清理机制：黑板会在每次游戏NPC请求指令时调用更新方法。该方法遍历当前的任务列表，并将最新的游戏环境数据传入任务目标的 is_satisfied() 方法进行检验。一旦检测到某项任务的设定目标在游戏环境中已被满足（例如目标容器中的物品数量已达到要求），黑板便会将该任务移除，从而防止智能体读取到已过期的需求。
	任务的过滤分发：每当需要构建智能体上下文信息的时候，黑板根据智能体能力信息和任务状态进行技能匹配（比对任务的 required_skill 字段与当前角色的技能列表）和前置条件校验（遍历任务的前置条件列表）两轮过滤。以此来降低智能体出现输出不符合自身能力或是不符合当前游戏环境的决策的可能性。
4.3	规划器
在基于大语言模型的智能体框架中，直接让大语言模型输出最底层的原子动作指令，会因为大语言模型对游戏内的机制、规则以及生产配方产生认知上的偏差，也就是常说的幻觉问题，而出现动作重复失败的问题。这不仅会增加额外的token消耗，还会降低智能体的表现从而破坏玩家的体验。
因此，本论文引入了规划器（Planner），实现了决策层与执行层的解耦，使得大语言模型能够从繁杂的底层物理规则、路径计算与配方约束中解放出来，专注于高层的意图决策与认知推理。这种设计的优势在于：
	促进涌现式行为： 允许大模型仅输出高层意图，使其能够专注于分析黑板任务、全局状态和自身属性，而不是关注于能不能拿取等繁杂的游戏规则，从而在多智能体交互中展现出更具逻辑性和不可预测的“涌现式行为”。
	规则的确定性处理： 将底层游戏引擎的硬性限制、物理规则和配方逻辑交由确定性的规划器来处理。规划器作为 LLM 与游戏引擎之间的逻辑沙箱，能够严格校验前置条件，弥补 LLM 在数值和规则推导上的短板。
	降低延迟和Token消耗：通过将常见的意图封装为确定的工作流（Workflow）并生成动作队列，智能体可以在较长的一段时间内自主执行任务，有效避免了对大模型的频繁轮询调用，大幅降低了 API 调用的 Token 消耗和网络延迟。
基于上述理念，规划器被设计为包含任务规划器（Task Planner）与行为规划器（Action Planner）的双层架构，分别实现高层意图的逻辑拆解与工作流的序列化。
4.3.1	任务规划器：基于GOAP的任务链拆解
任务规划器主要负责处理游戏世界中的确定性规则（如制作配方、前置资源校验等）。在这一层级，系统借鉴了GOAP的思想，将其扩展至多智能体协作层面。传统的单体 GOAP 会由单一智能体包揽所有前置动作，而在本系统中，任务规划器会将宏观任务拆解为子任务并投递至黑板，触发多智能体的协同效应。
首先，进行前置条件校验与递归拆解。当大模型决定执行高层意图时，任务规划器会先根据制作配方，对设施库存与全局库存进行检查，若发现原材料不足，系统会触发内部的供应链构建机制。该机制采用递归的思维，向上溯源所需材料的生产设施与次级原材料，自动生成对应的生产和搬运子任务。
其次，实现单次需求与总需求的解耦。在多智能体制造场景中，极易出现智能体等待所有资源齐备才开工的问题。为解决此痛点，规划器在生成任务链时，系统将任务的目标Goal设定为订单的总需求量，但将任务的前置条件放宽为仅需满足单次制作的用量。这意味着，只要全局或设施内有足以制作一个物品的材料，智能体即可开始局部执行，实现了搬运与生产流水线的并发运作，大幅提升了系统的并行效率。
最后，进行声明式任务的动态发布。 经过上述拆解出的所有子任务，均会被封装为黑板任务对象，内部包含明确的目标对象、前置条件与技能要求。这些子任务被统一下发至黑板任务池后，不同职业、具有不同技能的智能体会根据黑板的分发机制自主认领，从而共同完成原本单一智能体无法独立完成的复杂生产链。
4.3.2	行为规划器：意图的序列化
行为规划器负责将已经满足前置条件的高层意图（如搬运、进食、种植等）翻译为游戏引擎能够直接执行的原子动作队列（Action Queue）。其具体的机制如下：
首先，能够进行空间感知与动作序列化。 行为规划器在生成动作时，会动态评估智能体当前的位置状态与目标设施的空间关系。以搬运（Transport）意图为例，规划器会依次生成：向资源地移动（Move） -> 拿取物品（Take） -> 向目标容器移动（Move） -> 放入物品（Put）的指令链。同理，进食（Eat）意图会被序列化为寻找餐桌、移动、使用物品（Use）等原子指令序列。这种基于状态机的自动补全机制，保证了底层动作执行的逻辑连贯性。
其次，可以提供执行反馈与容错机制。 由于游戏环境是动态变化的，在动作执行期间，目标物品可能已被其他智能体拿走从而导致执行中断。因此，行为规划器在每一次序列化后都会返回一个包含执行状态信息的反馈封装对象（PlanResult）。如果规划成功，系统直接返回动作队列并流转至游戏引擎执行；如果规划失败（如资源突然不足、找不到目标设施），系统则会返回带有详细错误反馈的文本（如“资源缺失，系统已发布补货任务，请等待”），并向底层插入一个短时间的等待指令（Wait）以避免逻辑卡死。
然后，利用执行反馈指导大模型下一步决策。 上述规划失败的文本反馈并不会被系统丢弃，而是作为环境观察的一部分，在下一个时钟周期被感知器（Perceiver）重新捕获，并作为提示词（Prompt）的一部分再次输入给大语言模型。大语言模型接收到具体的错误反馈后，能够“意识到”当前任务已被挂起或遇到瓶颈，从而自主选择执行其他任务或进行休息。这一设计使得系统形成了一个完整的“观察-决策-执行-反馈”闭环。
